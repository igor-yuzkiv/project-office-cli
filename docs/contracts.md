# Project Office CLI — Контракти

Авторитетний опис контрактів і підходів проєкту `project-office-cli`. Самодостатній: усе, що
потрібно, щоб розуміти межі, інтерфейси й правила, — тут.

Мова: проза українською; ідентифікатори (назви команд, флагів, полів, ендпоінтів) — англійською.

---

## 1. Що це і межа

`project-office-cli` — **агент-орієнтований CLI**, контрольований шар між сесією AI-агента та
бекендом **Task Manager**. Веб-застосунок лишається інтерфейсом для людини; CLI — інтерфейс для агента.

- Агент працює **в межах одного проєкту** і дістається даних **лише через команди CLI** — без прямого
  доступу до бази, файлової системи чи внутрішньої структури застосунку.
- CLI спілкується з Task Manager **тільки через API**.
- Task Manager — окремий наявний бекенд (projects / task-lists / tasks / tags / comments); під CLI
  дороблюється окремий API-шар.

---

## 2. Трирівнева архітектура

| Рівень | Де | Що знає |
| --- | --- | --- |
| **Репозиторій** | `<repo>/.project-office/` | Локальні налаштування репо + `id`/ключ проєкту на бекенді. |
| **Бекенд** | Task Manager API | Проєкти й пов'язані сутності. Про репозиторії — **нічого**. |
| **Глобальний кеш** | `~/.project-office/` | Де лежать репо та з якими проєктами пов'язані; токен доступу; скретч-зона `tmp/`. |

CLI **склеює** дані бекенду з локальним кешем (напр., до деталей проєкту додає перелік репозиторіїв).

---

## 3. Доменна модель і скоуп

```
project                          межа роботи (посилання з repo-level .project-office)
 ├─ task-list                    групи в межах проєкту
 │   └─ task                     описова частина — структурований markdown
 │       ├─ comment              обговорення + усі текстові внески агента (поле kind)
 │       └─ tag[]                наскрізні мітки
 └─ repo[]  (ЛОКАЛЬНО)           зв'язок project↔repo живе в CLI, не на бекенді
```

- **Скоуп:** `project` — межа (агент не виходить за проєкт; примус — CLI-level, див. §7). `task` —
  фокус, передається явно на старті; агент може читати інші задачі *свого* проєкту.
- **Текстові внески агента** (note / feedback / proposal) → сутність `comment` з полем
  `kind` (`note` | `feedback` | `proposal`). Нових сутностей не заводимо.
- **Зміни стану** (створити задачу, статус, tag) → окремі команди-мутації, під гейтами.
- **Описи задач** — структурований markdown за шаблоном.
- **repos — локальні (per-machine):** «з яких репо складається проєкт» тримає CLI, не бекенд.

---

## 4. Контракт CLI↔агент: вивід

- **Бекенд завжди віддає чистий JSON**; **CLI — рендерер**.
- За замовчуванням CLI друкує **plain-текст під LLM**: компактний **frontmatter-заголовок з
  метаданими** + **markdown-тіло** (проходить наскрізь із бекенду). Прапорець **`--json`** віддає
  структуру як є.

```
AD-7 · In Progress · priority:high · @agent
Notifications: batch digest

<markdown-опис задачі>

--- comments ---
2026-07-04 @igor: подумай про rate-limit
```

- Перший рядок стабільний і машинно-локовний (`id · status · …`).
- `list` → один рядок-заголовок на елемент (без тіл); `view` → заголовок + повне тіло.
- **Plain — відповідальність CLI; бекенд лишається чистим JSON.**

---

## 5. Контракт CLI↔агент: ввід

- **Аргументи/флаги** (`commander`): позиційні — ідентифікатори (`task:view AD-1`); решта — флаги.
- **Великі / multi-line поля** (опис задачі, тіло коментаря):
  - головний механізм — **`--<field>-file <path>`** (агент пише файл, CLI читає байт-у-байт);
  - **stdin `-`** — вторинний (пайпи);
  - **inline** — лише короткі значення (`--status`, `--title`, `--kind`).
- **Скретч-зона `~/.project-office/tmp/`:** CLI туди скаффолдить шаблони й звідти читає. Файл
  видаляється **після успішного submit**; недороблені — по **TTL** або чистка на `install`.
- **Валідація на клієнті** перед відправкою → exit-код `2`.

---

## 6. Контракт помилок і exit-кодів

CLI друкує людино-читну помилку в `stderr`; **exit-код — детермінований сигнал агенту**.

```
0  ok                   2xx
2  validation           422   (message + errors по полях)
3  unauthenticated      401
5  forbidden / scope     403
4  not_found            404
7  conflict             409
6  backend_unavailable  5xx / мережа
1  generic              інше
```

---

## 7. Контракт CLI↔бекенд (API)

- **Транспорт:** REST/JSON поверх HTTPS; окремий namespace **`/api/cli/v1/…`**; версія в шляху.
  Авторизація — `Authorization: Bearer <token>`.
- **Конверт (Laravel-конвенції):**
  - успіх (ресурс): `{ data: { … } }`;
  - успіх (список): `{ data: [...], links, meta }`, параметри `page`, `per_page`;
  - помилка: `{ message, errors? }` + HTTP-статус (валідація — `422`).
- **Ідентифікатори:** задача — `key` `AD-1` (`projects.prefix` + `tasks.sequence_number`); проєкт —
  `prefix` у командах, ULID `id` канонічно в кеші. Внутрішні ULID агент не бачить.

**Ендпоінти (MVP):**
```
GET   /api/cli/v1/projects/:idOrPrefix
GET   /api/cli/v1/projects/:idOrPrefix/tasks?status=&list=&page=&per_page=
GET   /api/cli/v1/projects/:idOrPrefix/search?q=
GET   /api/cli/v1/tasks/:key
POST  /api/cli/v1/tasks
PATCH /api/cli/v1/tasks/:key
GET   /api/cli/v1/tasks/:key/comments
POST  /api/cli/v1/tasks/:key/comments
POST  /api/cli/v1/tasks/:key/tags     DELETE .../tags/:tag
```

## 8. Автентифікація та авторизація

- **Токен (PAT):** користувач сам створює токен у профілі Task Manager (login+password → токен).
  На бекенді — хеш. MVP: один широкий токен на всі проєкти користувача (поле `scope` закладаємо
  наперед під майбутні project-scoped токени).
- **Зберігання:** глобальний кеш `~/.project-office/` (права `600`), **не** repo-level. Токеном
  оперує **лише CLI** — агент **ніколи не бачить сирий секрет**.
- **Бутстрап — людський крок:** `install` відкриває Task Manager у браузері → користувач логіниться
  й створює токен → CLI зберігає в кеш. Не агент.
- **Межа проєкту (MVP):** токен широкий, **межу тримає CLI** (цілиться лише в прив'язаний проєкт);
  бекенд перевіряє, що user володіє проєктом. М'яка межа; тверда (backend-enforced) — на майбутнє.

## 9. Маршрутизація команд і структура коду

**Бінарник:** `project-office`. **Стиль:** `namespace:verb` (`task:view`, `comment:add`).

```
# setup / контекст
install                          бутстрап ~/.project-office/ + вхід
context                          показати зрезолвлений контекст (проєкт + репо)
# project
project:link <project>           прив'язати репо → проєкт
project:view                     деталі проєкту (+ репо з кешу)
# task
task:list [--status …]           список задач
task:view <AD-1>                 деталі задачі + коментарі
task:search <q>                  пошук у проєкті
task:new-draft                   заготовка опису за шаблоном у tmp/
task:create …                    створити задачу              (мутація, під гейтом)
task:edit <AD-1> [--status …]    змінити задачу/статус        (мутація, під гейтом)
# comment / tag / repo
comment:list <AD-1>              коментарі задачі
comment:add <AD-1> --kind …      текстовий внесок             (мутація)
tag:add|tag:remove <AD-1> <tag>  мітки                        (мутація)
repo:list                        репозиторії проєкту (локальний кеш, read-only)
```

**Структура `src/` (feature-sliced, дзеркало фронту):**
```
src/
  index.ts                # entry: commander, реєстрація namespace-ів
  entities/               # слайс на сутність: api/ render/ types/ (config/) index.ts
    project/ task/ comment/ tag/ repo/
  shared/                 # api-інфра (http-клієнт, ApiError), context-resolution,
                          # global-cache, tmp-scaffold, config (zod), lib (exit-коди)
  commands/               # тонкі хендлери; namespace → підпапка → файл на verb
```
**Межі:** `commands/` — лише оркестрація (context → api → render → exit-код); `entities/*/api` не
знає про рендер; `entities/*/render` не знає про HTTP.

---

## 10. Оборотні компроміси (MVP)

- repos — локальні, per-machine (§3).
- Текстові внески агента — у `comment` з `kind`, без нових сутностей (§3).
- Один широкий токен, межа проєкту — CLI-level (§8).

За потреби кожне з цих рішень розширюється, не ламаючи наявних інтерфейсів.
