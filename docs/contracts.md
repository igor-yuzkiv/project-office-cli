# Project Office CLI — Контракти

Цей документ фіксує базові контракти й межі `project-office-cli`.

Його задача — описати стабільні рішення: що таке CLI, де його межа, як він взаємодіє з агентом, бекендом і локальним середовищем.

Документ не є детальною технічною специфікацією API або повним описом реалізації. Конкретні endpoint-и, структура відповідей, фінальні сигнатури команд і деталі backend implementation визначаються окремими задачами на етапі планування.

Мова документа — українська. Назви команд, флагів, полів, сутностей і технічні ідентифікатори залишаються англійською.

---

## 1. Що це

`project-office-cli` — це агент-орієнтований CLI-шар між AI-агентом і бекендом Task Manager.

Веб-застосунок залишається основним інтерфейсом для людини. CLI не замінює web UI і не будується як другий повноцінний людський інтерфейс.

Його основна роль — дати агенту контрольований доступ до project-office контексту:

* читати проєкт і задачі;
* бачити повʼязані коментарі й теги;
* залишати текстові внески;
* у майбутньому — створювати або змінювати задачі через окремі контрольовані команди.

Агент не має працювати напряму з базою даних, внутрішньою файловою структурою Task Manager або приватною backend-логікою. Уся взаємодія має проходити через CLI і backend API.

---

## 2. Межа відповідальності

### Task Manager

Task Manager відповідає за доменні дані:

* users;
* projects;
* task lists;
* tasks;
* comments;
* tags;
* інші повʼязані backend-сутності.

Task Manager не має знати, де на конкретній машині користувача лежать репозиторії.

### Project Office CLI

CLI відповідає за агентський доступ і локальний контекст:

* визначає, у якому репозиторії запущений;
* визначає, до якого project привʼязаний репозиторій;
* читає глобальні локальні налаштування;
* звертається до Task Manager через API;
* рендерить відповідь у форматі, зручному для агента;
* контролює локальну межу проєкту.

### AI Agent

Агент взаємодіє з project-office тільки через CLI-команди.

Агент не повинен напряму читати або змінювати:

* backend database;
* backend storage;
* internal API implementation details;
* local CLI token storage;
* repo/project mapping files, якщо це не частина явно дозволеного flow.

---

## 3. Трирівнева модель даних

У системі є три рівні контексту.

| Рівень       | Де живе                   | Що зберігає                                                                       |
| ------------ | ------------------------- | --------------------------------------------------------------------------------- |
| Репозиторій  | `<repo>/.project-office/` | Локальну привʼязку repo до project.                                               |
| Backend      | Task Manager              | Проєкти, задачі, списки, коментарі, теги, користувачів.                           |
| Global cache | `~/.project-office/`      | Локальні налаштування CLI, token, список repo/project звʼязків, scratch/tmp зону. |

Repo-level config описує тільки поточний репозиторій.

Global cache описує локальну картину на машині користувача.

Backend залишається джерелом правди для доменних сутностей, але не для локальних repo-шляхів.

CLI може склеювати backend-дані з локальним кешем. Наприклад, команда перегляду project може отримати project із backend і додати до виводу список локально відомих repo, які привʼязані до цього project.

---

## 4. Доменна модель на рівні CLI

Базова модель:

```text id="u7616c"
project
 ├─ task-list
 │   └─ task
 │       ├─ comment
 │       └─ tag[]
 └─ repo[]  local only
```

`project` — основна межа роботи агента.

`task` — основний робочий фокус.

`comment` — місце для текстових внесків агента.

`tag` — наскрізна мітка для класифікації задач.

`repo` — локальна привʼязка репозиторію до project. На MVP ця інформація не є backend-сутністю.

---

## 5. Текстові внески агента

Агентські нотатки, фідбек і пропозиції не створюють окремих нових сутностей.

Для цього використовується існуюча сутність `comment`.

Коментар може мати додаткову класифікацію, наприклад:

```text id="ssg70a"
note
feedback
proposal
```

Точний набір значень і спосіб зберігання визначається на етапі реалізації comments flow.

Головний принцип: усе, що агент залишає як текстовий внесок, має бути видно людині у звичному web UI Task Manager.

---

## 6. Зміни стану

Read-only команди можуть виконуватись напряму.

Команди, які змінюють стан системи, мають розглядатись окремо й можуть вимагати gate або підтвердження.

До мутацій належать:

* створення задачі;
* редагування задачі;
* зміна статусу;
* додавання або зняття tag;
* створення comment;
* інші дії, які змінюють backend state.

Правила gates не фіксуються в цьому документі детально. Вони мають бути визначені окремо, коли зʼявиться перший write-flow.

---

## 7. Контракт CLI ↔ Agent: output

Backend повертає структуровані дані.

CLI відповідає за рендер відповіді для агента.

За замовчуванням CLI має віддавати компактний plain-text/markdown вивід, який легко читати агенту.

Орієнтовний формат:

```text id="h0g7jy"
AD-7 · In Progress · priority:high

<markdown task description>

--- comments ---
2026-07-04 @igor: note text
```

Основна ідея:

* `list` команди повертають короткий список без великих тіл;
* `view` команди повертають деталі;
* markdown-описи задач проходять наскрізь без зайвого переформатування;
* за потреби має бути машинний режим на кшталт `--json`.

Точний формат кожної команди може уточнюватись під час реалізації, але принцип залишається: backend не рендерить agent-friendly text, це відповідальність CLI.

---

## 8. Контракт CLI ↔ Agent: input

CLI приймає короткі значення через аргументи й флаги.

Приклади:

```text id="hq4qmz"
task:view AD-1
task:list --status in-progress
comment:add AD-1 --kind note
```

Великі або multi-line значення не варто передавати inline.

Для великих текстів основний механізм:

```text id="bqd3cj"
--<field>-file <path>
```

Наприклад:

```text id="e6v9zv"
comment:add AD-1 --kind proposal --body-file ./proposal.md
```

Додатково може підтримуватись stdin через `-`, якщо це буде зручно для агентського flow.

Inline-значення варто залишати тільки для коротких полів:

* status;
* title;
* kind;
* tag;
* key;
* id/prefix;
* прості фільтри.

---

## 9. Scratch / tmp зона

CLI може використовувати локальну scratch-зону в global cache.

Орієнтовне призначення:

* тимчасові markdown-заготовки;
* draft-файли для створення задач;
* файли для довгих comment body;
* проміжні артефакти, які не мають потрапляти в repo.

Очікуване місце:

```text id="ex3mzk"
~/.project-office/tmp/
```

Файли, які були успішно відправлені через CLI, можуть видалятись автоматично.

Старі або недороблені tmp-файли можуть чиститись за TTL або окремою службовою командою.

Точні правила cleanup визначаються під час реалізації CLI.

---

## 10. Помилки й exit codes

CLI має повертати передбачувані exit codes, щоб агент міг розуміти результат без парсингу довгого тексту.

Базові категорії:

```text id="e5bj9a"
0  ok
1  generic error
2  validation / input / config error
3  unauthenticated
4  not found
5  forbidden / scope error
6  backend unavailable
7  conflict
```

CLI друкує людино-читну помилку в `stderr`.

Точне маплення backend HTTP errors на exit codes може уточнюватись на етапі реалізації API client, але ці категорії вважаються стабільним контрактом для агента.

---

## 11. Контракт CLI ↔ Backend

CLI спілкується з Task Manager тільки через HTTP API.

Backend API для CLI має бути окремо усвідомленим шаром, а не випадковим використанням внутрішніх web routes.

На цьому етапі не фіксується:

* точний URL namespace;
* versioning API;
* фінальний список endpoint-ів;
* повна структура response DTO;
* назви controller/action;
* точний формат query parameters.

Фіксуються тільки принципи:

* API має бути захищене bearer token authentication;
* API має повертати JSON;
* API має працювати в межах користувача, який створив token;
* backend має перевіряти доступ користувача до requested project/task;
* CLI не має бачити внутрішні backend-only identifiers без потреби;
* відповіді мають бути достатньо стабільними, щоб CLI не залежав від web UI implementation.

Конкретні endpoints будуть визначатись окремими задачами під конкретні CLI-команди.

---

## 12. Ідентифікатори

Для агента основним ідентифікатором задачі має бути human-readable key, наприклад:

```text id="f5bfi2"
AD-1
AD-27
```

Точне формування key залежить від поточної моделі Task Manager.

Внутрішні backend identifiers можуть використовуватись CLI або backend, але не мають бути основним інтерфейсом для агента, якщо є стабільний людський key.

Для project на рівні CLI може використовуватись короткий key/prefix або інший зрозумілий ідентифікатор. Канонічне рішення визначається під час реалізації project linking flow.

---

## 13. Автентифікація

Користувач створює personal API token у web UI Task Manager.

Token використовується CLI для звернення до backend API.

Базові принципи:

* token створює людина, не агент;
* token показується користувачу тільки один раз після створення;
* backend не зберігає plain token;
* CLI зберігає token у global cache;
* token не зберігається в repo-level `.project-office`;
* агент не має отримувати сире значення token у виводі команд.

На MVP достатньо personal token, який діє в межах доступу користувача.

Project-scoped tokens, abilities/scopes і складні permission rules не фіксуються як обовʼязкова частина першої версії.

---

## 14. Локальна межа проєкту

На MVP project boundary частково контролюється CLI.

Repo-level config каже CLI, до якого project належить поточний репозиторій.

CLI-команди мають за замовчуванням працювати саме в цьому project context.

Backend зі свого боку має перевіряти, що authenticated user має доступ до requested project/task.

Більш жорстка backend-enforced project boundary через project-scoped tokens може бути додана пізніше.

---

## 15. Команди

CLI використовує стиль команд:

```text id="ax1nu0"
namespace:verb
```

Приклади майбутніх команд:

```text id="3ufrmq"
project:view
task:list
task:view
task:search
comment:list
comment:add
tag:add
tag:remove
repo:list
```

Цей список не є фінальним API контрактом.

Фінальні сигнатури команд, обовʼязкові аргументи, флаги й поведінка визначаються окремо під час планування кожного command flow.

Стабільним вважається тільки загальний стиль: короткі, явні команди, які відображають дію над сутністю.

---

## 16. Структура CLI-коду

CLI-код має бути розділений так, щоб command handlers не перетворювались на “кухню всього”.

Базовий принцип:

* command handler оркеструє дію;
* API layer відповідає за backend calls;
* render layer відповідає за output;
* config/context layer відповідає за локальний context;
* validation/error layer відповідає за передбачувану поведінку CLI.

Точна структура папок не фіксується як контракт.

Орієнтовно можна рухатись у feature-sliced стилі, близькому до frontend-підходу, але фінальне розбиття має визначатись під час реалізації CLI skeleton.

---

## 17. Оборотні компроміси MVP

У першій версії свідомо допускаються такі компроміси:

* repo/project mapping живе локально;
* personal token широкий у межах доступу користувача;
* agent notes/proposals живуть у comments;
* точні gates для мутацій визначаються пізніше;
* CLI API формується поступово під реальні команди;
* повний shared registry репозиторіїв на backend не реалізується.

Ці рішення мають бути оборотними. Їх можна буде посилити або винести в backend без ламання базової ідеї CLI.
