# Огляд проєкту (українською)

Цей документ пояснює, що таке `project-office-cli`, як він влаштований і як усе працює
разом — простою мовою, без припущення, що читач одразу лізе в код. Технічні деталі кожної
частини описані в окремих файлах цієї теки (`docs/`) — тут дається карта, як усе
поєднується.

## 1. Що це за проєкт і навіщо він потрібен

`project-office-cli` — це командний інтерфейс (CLI), через який AI-агенти працюють з
Project Office / MVP Task Manager (застосунком для керування задачами).

Головна ідея: **агент ніколи не працює з базою даних, бекендом чи файловою системою
застосунку напряму.** Єдиний дозволений шлях — ця CLI. CLI сама звертається до бекенду
через API, а агенту віддає вже готовий, зрозумілий, безпечний вивід.

Є ще одна CLI-подібна річ поруч — `backlog` (Project Office / Backlog.md), яка керує
задачами _самого процесу розробки_ (той офіс, де зараз лежить ця задача TASK-0015). Це
**інший інструмент** і не має нічого спільного з `project-office-cli` — не плутати.

Людський інтерфейс до Task Manager — окремий веб-застосунок. `project-office-cli` — це
паралельний, агентський вхід у ті самі дані, з іншими правилами доступу.

## 2. Як CLI розуміє, з яким проєктом вона працює

Кожна команда (крім `install` і двох спеціальних команд нижче) працює в межах **одного,
наперед визначеного проєкту**. Агенту не потрібно щоразу передавати `--project` — CLI сама
визначає це з контексту.

### 2.1 Локальний маркер репозиторію

Все базується на одному файлі:

```txt
<репозиторій>/.project-office/repo-settings.json
```

Цей файл каже: "цей репозиторій прив'язаний до проєкту з таким-от `projectId`". Коли ти
запускаєш будь-яку команду, CLI піднімається вгору по теках від поточної директорії, шукає
цей файл, читає з нього `projectId` — і саме він потім підставляється в усі запити до
бекенду.

**Важливо:** цей файл ніколи не редагується вручну. Його пише виключно одна команда —
`project:link-repo` (див. нижче). Якщо агент бачить, що файла немає або він битий —
правильна дія: викликати `project:link-repo`, а не створювати файл самому.

### 2.2 Локальний кеш проєкту (опційний)

Окремо, поза репозиторієм, CLI може тримати локальний кеш-файл на диску користувача:

```txt
~/.project-office-cache/projects/<projectId>.json
```

Це **не обов'язкова** річ — команди працюють і без неї. Це просто зручний офлайн-знімок:
дані про сам проєкт (назва, статус тощо) плюс список репозиторіїв, які до нього прив'язані
(`repos[]`). Кожен запис у `repos[]` — це такий самий об'єкт, що й пишеться у
`repo-settings.json` (`projectId`, `path`, `name`, `description`, `stack`) — тип
`ProjectRepositoryDefinition`. Файл, що містить снапшот проєкту + список репо, називається
`ProjectRegistryRecord`.

### 2.3 Дві команди, які "запускають" цей механізм

- **`project:connect --project <id>`** — іде на бекенд, забирає дані проєкту, створює або
  оновлює його кеш-файл. **Ніколи** не чіпає `repo-settings.json`. Якщо кеш вже був — просто
  оновлює знімок проєкту, а список репо (`repos[]`) не чіпає.
- **`project:link-repo --project <id> --name <назва>`** — робить те саме, що `connect`
  (якщо кеш-файла ще нема), **плюс** дописує поточний репозиторій у `repos[]`, **плюс**
  записує `repo-settings.json` у цьому репозиторії. Це єдина команда, яка має право писати
  цей файл. Повторний виклик для того самого шляху — не дублює запис, а заміщує його
  (ідемпотентно).

Типовий сценарій підключення нового репо до проєкту:

```bash
project-office install                # один раз — налаштувати токен доступу
project-office project:link-repo --project 01k... --name my-repo
# тепер усі інші команди в цьому репо самі знають свій projectId
```

### 2.4 Хто за що відповідає в коді

Уся ця логіка живе в `src/shared/libs/project-office/`, розділена на дві частини за
принципом "читання окремо від запису":

- **`selectedProjectContext`** (`selected-project-context.ts`) — тільки читає. При старті
  CLI один раз викликає `bootstrap()`, який шукає `repo-settings.json` і, якщо там є
  `projectId`, ще й підвантажує кеш проєкту. Звідси команди дістають `getProjectId()`.
- **`projectsRegister`** (`projects-register.ts`) — тільки пише. Уміє: прочитати/записати
  кеш проєкту (`getProjectRecord` / `setProjectRecord`), додати репо в кеш
  (`linkRepositoryToProject`), записати `repo-settings.json`
  (`writeRepositorySettings`).

Детальніше — [Project Office context](./project-office-context.md) (англійською, технічний
рівень).

## 3. Формат виводу команд

Кожна команда, яка щось повертає, підтримує опцію `-f, --format`:

- **`markdown`** (за замовчуванням) — блок властивостей у форматі YAML (`---\n...\n---`),
  а після нього — звичайний текст або список. Зроблено для того, щоб агенту було зручно
  одразу читати відповідь, не парсячи JSON.
- **`json`** — сира відповідь від бекенду, форматований JSON. Потрібен, коли агенту треба
  щось спарсити програмно.

Значення в markdown-блоці властивостей беруться в лапки (`key: "value"`), масиви
рендеряться як YAML-список:

```yaml
tags:
    - cli
    - infra
```

Це не повноцінний YAML-серіалізатор (без екранування спецсимволів понад базові лапки) — це
свідомий компроміс на користь простоти. Технічні деталі — [Output rendering](./output-rendering.md).

Немає жодного диспетчера форматів — кожна команда сама вирішує, який рендер викликати
(`if (format === 'markdown') {...} else {...}`). Це навмисно: простіше й прозоріше, ніж
абстрактна фабрика заради двох варіантів.

## 4. Огляд команд

### Читання (без побічних ефектів)

| Команда         | Що робить                                                                    |
| --------------- | ---------------------------------------------------------------------------- |
| `project:view`  | Показує поточний проєкт разом зі списком локально прив'язаних репо (з кешу). |
| `task:list`     | Список задач (без пошуку/фільтрів).                                          |
| `task:search`   | Пошук задач за текстом (`--query`).                                          |
| `task:view`     | Одна задача за id/ключем.                                                    |
| `task:comments` | Список коментарів до задачі.                                                 |

### Запис (змінюють дані на бекенді)

| Команда               | Що робить                                                                                      |
| --------------------- | ---------------------------------------------------------------------------------------------- |
| `task:create`         | Створює задачу (назва, опис, теги через кому).                                                |
| `task:update`         | Змінює назву, статус, опис і/або теги задачі.                                                 |
| `task:comment-add`    | Додає коментар до задачі.                                                                      |
| `task:comment-update` | Редагує наявний коментар.                                                                      |

Довгий текст (`--description`, `--content`) можна передати трьома способами: як звичайний
рядок, як `@шлях/до/файлу` (тоді CLI прочитає файл), або як `-` (тоді CLI прочитає
`stdin`) — зручно для довгих описів у скриптах.

### Налаштування контексту

| Команда             | Що робить                                                                                |
| ------------------- | ---------------------------------------------------------------------------------------- |
| `install`           | Одноразове налаштування: токен доступу + тека кешу.                                      |
| `project:connect`   | Забирає проєкт з бекенду, оновлює лише локальний кеш.                                    |
| `project:link-repo` | Прив'язує поточний репозиторій до проєкту (єдина команда, що пише `repo-settings.json`). |

### Службові

| Команда                  | Що робить                                                                             |
| ------------------------ | ------------------------------------------------------------------------------------- |
| `instructions [command]` | Друкує агентську інструкцію — або огляд усього CLI, або по конкретній команді.        |
| `debug`                  | Внутрішня команда для локальної розробки самого CLI, не для реальної роботи з даними. |

## 5. Як CLI спілкується з бекендом

Усі запити йдуть через один спільний HTTP-клієнт (`src/shared/libs/http/`). Він сам додає
заголовок авторизації (`Bearer <токен>`) до кожного запиту — токен береться з локальних
налаштувань, які `install` зберігає у `~/.project-office-cache/settings.json`.

Якщо запит падає, клієнт перетворює технічну помилку axios на просте, безпечне повідомлення
(`HttpError`) — без витоку токена чи внутрішніх деталей запиту в лог чи вивід. Деталі —
[HTTP client](./http-client.md).

## 6. Дві документації, і чому вони окремі

У проєкті свідомо існують **два** джерела документації по командах:

1. **`docs/commands/*.md`** (цей каталог) — людська документація. Пишеться і оновлюється
   як окрема фаза після реалізації задачі, лише з дозволу користувача.
2. **`src/commands/instructions/content/*.md`** — агентська документація, яку сама CLI
   віддає командою `instructions`. Вона вшита прямо в скомпільований бінарник (через
   `import ... with { type: 'text' }`), тому працює навіть без доступу до цього репозиторію
   у рантаймі. Тон і структура — під агента (точний виклик, семантика опцій, форма виводу,
   типові помилки), а не під людину, яка читає документацію в браузері.

Коли додається чи змінюється команда — обидва джерела мають лишатися синхронними з реальною
поведінкою команди, але оновлюються незалежно одне від одного.

## 7. Структура коду (коротко)

```txt
src/
  commands/   # реєстрація CLI-команд (одна тека на команду/групу)
  entities/   # доменні сутності: task, project, tag, user
  shared/     # спільна інфраструктура: http, output, project-office, config, utils
```

- `entities/<name>/api/` — функції запитів до бекенду (`fetch...Request`,
  `create...Request`, …).
- `entities/<name>/render/` — сервіси, які перетворюють сутність на markdown-вивід.
- `entities/<name>/types/` — типи сутності.
- `shared/libs/output/` — генеричні `renderJson`/`renderMarkdown`, без знань про конкретну
  сутність.
- `shared/libs/project-office/` — усе про контекст проєкту (розділ 2.4 вище).
- `shared/config/` — фіксовані значення конфігурації (не секрети — ті в `.env`).

Повніше — [architecture.md](../.claude/rules/architecture.md) та
[conventions.md](../.claude/rules/conventions.md) (правила проєкту для агента, англійською).

## 8. Швидкий старт (для розуміння сценарію "з нуля")

```bash
# 1. Одноразово: налаштувати токен доступу і кеш-теку
project-office install

# 2. Прив'язати поточний репозиторій до проєкту
project-office project:link-repo --project 01k... --name my-repo

# 3. Далі — звичайна робота, projectId підставляється сам
project-office task:list
project-office task:view --task TASK-1
project-office task:create --name "Нова задача" --tags "bug,backend"

# 4. Якщо потрібна підказка, як викликати команду
project-office instructions task:create
```
